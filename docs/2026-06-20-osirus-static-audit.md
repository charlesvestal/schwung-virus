# Osirus (Access Virus / gearmulator) Schwung Module — Static Audit

Primary file: `/Users/josh/schwung repos/schwung-virus/src/dsp/virus_plugin.cpp` (parent + forked-child DSP, shm IPC).
Submodule: `/Users/josh/schwung repos/schwung-virus/libs/gearmulator/.../microcontroller.cpp`.

**Audit thesis (the bug class being hunted):** `set_param` writes one shm field (user intent) while `get_param`/`get_state` read a *different, lagging* field (what the child currently runs), so a just-set value reads back stale and fails to survive the host's autosave→restore. The recently-fixed `dsp_clock` `percent`-vs-`applied` split is the canonical instance. I traced every settings/synthparam key, the state serialize/restore paths, the child-restart lifecycle, and the preset/bank browser path for the same pattern.

**Headline:** The uncommitted `dsp_clock` read fix is **complete and symmetric** (both read sites prefer `dsp_clock_percent`); no other settings key (`gain`, `octave_transpose`, `rom_index`, `preset`, `bank_index`) exhibits the split. The audited prior fixes (#14, #15, 65dc5dc, 029b9ea, e4aaf60) are all **intact in current code and not undermined** by later edits. The remaining items are all **low-severity, narrow-window or cosmetic**, not data-loss or stuck-patch bugs. There is **one structurally-identical-to-`dsp_clock` field**, `cur_bank_preset_count`, that the parent reads but cannot keep current — self-healing in practice, but worth closing.

## (A) Confirmed bugs to fix

Ranked by severity × confidence. All are LOW severity (none cause silence, lost patches, or stuck wedges); they are latent/narrow-window or cosmetic.

### A1. `cur_bank_preset_count`: parent bounds `set_param("preset")` against a child-owned field it never updates (dsp_clock-class asymmetry)
- **Where:** write side `set_param("bank_index")` at `virus_plugin.cpp:2195-2222` (never writes `cur_bank_preset_count`); read side `set_param("preset")` at `virus_plugin.cpp:2176-2177` (`int pmax = shm->cur_bank_preset_count > 0 ? shm->cur_bank_preset_count : shm->preset_count;`); the only writer is the **child** at `:1143` (`shm->cur_bank_preset_count = pc_in_bank;`); seeded to global count at `:1531`. Same lagging field also read by `get_param("preset_count")` `:2611` and `get_state` `:2722`.
- **Bug:** Exact dsp_clock pattern — parent reads a field only the child updates. After switching from a large ROM bank to a smaller user bank, there is a multi-ms window (until the child drains the bank-select from the MIDI FIFO) where the parent still bounds against the OLD bank's larger count. A preset index out-of-range for the new bank passes the parent guard; `get_param("preset_count")` polled in that window also reports the stale larger count.
- **Why low, not medium:** Self-healing and never strands a patch. The child re-bounds (`:1140-1142` clamp to `pc_in_bank-1`) and falls back to preset 0 on a failed load (`:1162-1168`), then `get_param("preset")` (`:2609`) returns the corrected value. Normal interactive use (human picks a preset seconds after a bank change) never hits the window; only rapid programmatic bank+preset bursts trip it.
- **Fix (important correction to the original suggestion):** The originally-proposed "have the parent compute the new bank's count up front" is **NOT viable** — verified: `g_user_banks`/`g_user_bank_count`/`g_rom_bank_count` (`:684-686`) are populated only in the forked child (`child_load_user_banks` at `:1295`, called at `:1537`; `g_rom_bank_count` set at `:1529`, both inside the child branch). In the parent these are 0 (which is also why the parent's `bank_index` `bank_name` snprintf at `:2208-2213` always falls to the generic `"Bank %d"` else-branch — a separate latent cosmetic nit). Two viable fixes instead:
  1. **Simplest:** make `set_param("preset")` bound only against the global `shm->preset_count` and rely entirely on the child's re-bound + fallback — removing the parent's dependence on a field it cannot keep current. (Lowest risk.)
  2. **Or:** have the child publish a per-bank preset-count table into shm that the parent can index by `current_bank`.

### A2. Legacy/embedded-single restore bounds preset by global `preset_count`, not the bank's actual count
- **Where:** legacy state branch `virus_plugin.cpp:2131` (`ival < shm->preset_count`); embedded-single label sync `:2092` (same); contrast the live setter `:2176` which uses the bank-specific `pmax`.
- **Bug:** Restoring into a user bank shorter than the global count can pass the parent's bound with an out-of-range index. The child re-clamps/falls back (`:1140-1142`, `:1162-1168`), so the patch is never stranded, but `current_preset` momentarily names an out-of-range slot and the readback label (`:2611`, `:2722`) can disagree with what the child loaded.
- **Reachability is narrow:** the legacy branch (`:2120-2170`) only runs when the saved state has NO embedded `single` (modern self-contained saves return early at `:2116`), so the audio-affecting path is legacy-only. The `:2092` embedded-single sync is on the modern path but is explicitly cosmetic (comment `:2088-2089` — "the audible patch comes from the embedded single") and issues no PC.
- **Fix:** Bound the restored preset by the **restored** bank's real count. Note: do **not** read `cur_bank_preset_count` at `:2131` — it is stale there (bank was set at `:2123` but the field is never recomputed for the new bank). Compute `bank_preset_count(shm, restored_bank)` directly (helper at `:1041-1047`). Apply to both `:2092` and `:2131`. (Overlaps A1 — fixing A1 via option 1 plus this makes the bound consistent everywhere.)

### A3. Self-contained restore does not bump `preset_req_gen`, so a `get_state` in the inject window can serialize the OLD single / OLD per-params into autosave
- **Where:** self-contained restore bumps only `shm->pending_single_req_gen++` (`virus_plugin.cpp:2096`) and returns (`:2116`), never touching `preset_req_gen`. The child applies the inject asynchronously in `child_handle_single_io` (`:1060-1071`), writing `current_single` + `current_single_valid=1` (`:1066-1067`) and refreshing `cc_values`/`cc_seen` via `child_sync_params_from_preset` (`:1065`). But `get_state`'s blocking wait keys only on the preset generation (`:2682-2687`, `for(... shm->preset_done_gen != req ...)`), which the self-contained branch never advances — so the wait returns immediately and does NOT wait for the inject.
- **Two related consequences (same root):**
  - **Per-param tail** (`:2740-2744`) can emit the PREVIOUS patch's `cc_values` (read via `get_param_value` `:802-806`). This tail is a documented best-effort remote-UI seed (comments `:2724-2729`); recall itself is unaffected because restore consumes the embedded `single` hex.
  - **Embedded single** (`:2730-2735`, serializes `shm->current_single`) can serialize the OLD `current_single` bytes if `current_single_valid` was already TRUE from a prior patch — i.e. a restore-over-an-already-loaded-patch followed by an instant autosave could persist the previous patch. On the common startup autosave→restore the child is freshly forked and `current_single_valid` is 0, so `get_state` hits the EMPTY bailout at `:2695-2698` and the host's `bailIfEmpty` preserves the good slot — no clobber. Only restore-over-valid is at risk.
- **Why low:** sub-millisecond window; restore is normally followed by playback, not an instant autosave; the common case is protected by the empty-state guard.
- **Fix (mirror the dsp_clock lesson — make the read wait on the write it depends on):** in `get_param("state")`, add a second bounded wait on `shm->pending_single_done_gen == shm->pending_single_req_gen` before serializing, OR have the self-contained restore branch also bump `preset_req_gen` (and the child set `preset_done_gen` after consuming `pending_single`). This closes both the per-param and embedded-single windows at once.

### A4. `current_single_valid` is never reset, so it stays stale-TRUE across a ROM-switch child restart (stale-single / mismatched-rom_index window)
- **Where:** `current_single_valid` is assigned `1` only at `virus_plugin.cpp:1067` (inject) and `:1174` (Program Change); **no write of 0 exists anywhere** (decl `:663`). `kill_child_and_reset` (`:1762-1781`) resets ring/midi/handshake/profiling fields and `dsp_clock_applied=0` (`:1780`) but never touches `current_single_valid` or `current_single[]`. The new child's boot deliberately does not set valid (comment `:1085-1090`). The shm is a single persistent mmap reused across restarts (`:1829`, comment `:1747`).
- **Bug:** After any real patch load, valid stays 1 through a ROM switch while `current_single[]` still holds the PREVIOUS ROM's 512 bytes and `rom_index` has already been updated to the NEW value (`:2240` explicit setter / `:2060` restore path). A `get_state` in the restart window skips the empty guard (`:2695`) and serializes old-ROM single bytes under the new `rom_index` (and, since `percent==0` and `applied==0` after reset, `dsp_clock:40`, plus the OLD `rom_model_name` which the child updates only post-fork at `:1524`) — an internally inconsistent state. Unlike a stale-empty read, this non-empty read is NOT protected by `bailIfEmpty` and CAN clobber a good slot file; if no later good save intervenes, a host restore injects old-ROM bytes into the new-ROM child.
- **Why low:** ≥1s window during an intentional, infrequent ROM switch; requires an autosave to land in it; the truly-mismatched bytes survive only a few ms (the passive refresh at `:1084`, gated on the sticky valid flag, overwrites `current_single` with the new child's default within ~8 emu iterations), and live state self-heals via the `pending_state` replay (`:2811-2817`). The disk-clobber is the residual risk.
- **Fix:** In `kill_child_and_reset`, add `shm->current_single_valid = 0;` (and optionally zero `current_single`) alongside the existing transient resets at `:1780`. The empty-state guard then protects the ROM-switch window exactly as it protects a fresh boot; `pending_state` replay re-establishes `valid=1` with the new ROM's single. (This is the documented invariant — comments `:1088-1090`, `:2693-2694` — currently violated only because the flag is sticky.)

### A5. After self-contained restore, clearing one force-axis leaves the other latched → spurious same-index bank reselect loads preset 0
- **Where:** restore sets both `inst->force_next_preset = 1; inst->force_next_bank = 1;` (`virus_plugin.cpp:2113-2114`). Each guard clears only its own flag: preset `:2180`/`:2184`, bank `:2200`/`:2204`. No other code clears either flag.
- **Bug:** After restore, if the user reselects the same-index PRESET (clears `force_next_preset`, makes the edit buffer authoritative), `force_next_bank` stays latched. A later same-index BANK reselect then bypasses the no-op guard at `:2200` and issues `CC32 + PC 0` (`:2216-2219`) plus `current_preset=0` (`:2207`) — an audible jump to preset 0 of the already-current bank instead of a no-op. (Note: this is an actual patch change to preset 0, not merely a redundant reload, unless the current preset was already 0.)
- **Why low:** Narrow trigger chain (self-contained restore → same-index preset reselect → same-index bank reselect); cosmetic/UX surprise, never silence or a wedge. The e4aaf60 per-axis split is correct for its stated goal; this is a secondary edge it doesn't cover.
- **Fix:** Once any real PC is pushed (edit buffer authoritative on both axes), clear BOTH flags. Cheapest: in the preset set path (`:2184-2190`) and bank set path (`:2204-2219`) set `inst->force_next_preset = inst->force_next_bank = 0;` together.

### A6. Non-self-contained restore with `bank` but no `preset` updates the label but issues no Program Change
- **Where:** `virus_plugin.cpp:2122-2130` sets `current_bank`/`bank_name` unconditionally; the `CC32+PC` pair is emitted only inside `if (has_preset)` at `:2135-2146`. The per-param overlay (`:2158-2165`) and octave/dsp_clock/gain still run.
- **Bug:** For a state with `bank` but no `preset`, the browser's bank label and `current_bank` advance but the child is never told to load from that bank — the audible patch stays the old edit buffer (optionally param-overlaid) while the UI claims the new bank.
- **Why low:** Only reachable from legacy/externally-authored state JSON. This module's own `get_state` always emits both keys (`:2703-2704`) and prefers the self-contained `single` branch (`:2085-2117`), so self-produced states are immune.
- **Fix:** When `bank` is present but `preset` is missing, treat preset as 0 and emit the `CC32+PC 0` pair, matching `set_param("bank_index")` (`:2207`, `:2216-2219`).

### A7. Boot/restart window: auto-mode `dsp_clock` read reports the literal `40` instead of the model default
- **Where:** the fallback chain `... applied > 0 ? applied : 40` at `get_param` `:2627` and `get_state` `:2710-2711`. `kill_child_and_reset` zeroes `dsp_clock_applied` (`:1780`); the new child re-writes it only at `:1521`.
- **Bug:** While the new child boots in auto mode (`percent==0`, `applied==0`), reads hit the literal `40`, which is no model's default (A=100, B=45, C=35). An autosave in this window persists `dsp_clock:40`. On restore, `apply_restored_dsp_clock(40)` floors it: A (40<100→auto), B (40<45→auto), but **C (40≥35→kept 40)** — so a Virus-C restart-window autosave can pin the clock to 40 instead of 35.
- **Why low:** Requires the conjunction of auto mode + autosave landing in the ~1-2s restart window + Virus C; a 40% vs 35% CPU budget difference, no audio loss; `get_param`/`get_state` agree (both 40), no cross-read asymmetry.
- **Fix:** Make the auto fallback model-aware (derive from `rom_model_name` like `apply_restored_dsp_clock`), or emit `0` to signal auto so the reader/child re-derives, instead of the magic `40`.

### A8 (trivial). Stale shm comment: `dsp_clock_percent` claims B/C auto-default is 50; code uses B=45 / C=35
- **Where:** `virus_plugin.cpp:636` — `/* 0 = auto (100 for A, 50 for B/C) */`. Actual child defaults `:1513-1518` are A=100/B=45/C=35; parent floor guard `:2023-2026` matches. No `50` anywhere in clock logic.
- **Fix:** Update the comment to `/* 0 = auto (100 for A, 45 for B, 35 for C/others) */`. Pure doc drift; risk is misleading a future editor toward 50, desyncing from the floor guard.

## (B) Regression risks / latent foot-guns (no fix required now)

### B1. `get_state` param loop lacks the `off < buf_len` backstop its siblings have (defense-in-depth)
- **Where:** `virus_plugin.cpp:2740-2744` (and the closing `}` at `:2745`). Sibling loops all guard: `:2644` (`off < buf_len - 80`), `:2769` (`- 200`), `:2780` (`- 50`).
- **Risk:** `snprintf` returns the would-have-written length, so `off` can grow past `buf_len`; the next call passes a negative `int` as `size_t` (~SIZE_MAX) → OOB write. **Not reachable today:** `buf_len = SHADOW_PARAM_VALUE_LEN = 65536`, worst-case output ~5-7KB. Becomes live the moment a smaller buffer is passed or the param/value set grows ~10x.
- **Fix:** Add `&& off < buf_len - 64` to the loop and clamp after each snprintf (`if (off >= buf_len) { off = buf_len - 1; break; }`); same clamp on the closing `}`.

### B2. `bank` serialized twice — as `"bank"` and `"bank_index"`; restore reads only `"bank"`
- **Where:** `get_state` emits `"bank"` at `:2703` and `"bank_index"` at `:2720`, both from `shm->current_bank`. Restore reads only `"bank"` (`:2090`, `:2122`); no `json_get_int(..., "bank_index", ...)` exists. The `"bank_index"` emission is an intentional remote-UI seed (comment `:2714`).
- **Risk:** Currently symmetric and harmless. Foot-gun only if a future edit re-keyed restore off `"bank_index"` or let the two emissions diverge.

### B3. `dsp_clock` setter clamps to [10,100] but not to the per-model floor
- **Where:** setter `:2252-2256`. The restore guard `apply_restored_dsp_clock` (`:2019-2035`) catches sub-floor values at the restore seam, so this is not a persistence bug — but a user can momentarily run a starving sub-floor clock live (e.g. 50 on Virus A) until restore. Optional hardening, not a regression.

**CR-1 — Child DSP crash is permanently unrecoverable and silent**

### G1. Crash recovery / child liveness watchdog (HIGH severity, NEW)

### G2. Cross-process memory ordering on the audio ring and MIDI FIFO index publish (MEDIUM, NEW)

The audio-ring producer in the child writes all sample slots then publishes the new index with a plain store and NO barrier: shm->audio_ring[...] writes in the loop at 1672-1676 followed immediately by shm->ring_write = wr at 1677 (same pattern in the prefill at 1568-1580). The parent consumer reads ring_write (via shm_ring_available, 692-696) and then reads the sample slots (2822-2832), also with no acquire barrier. The MIDI FIFO producer is identical: midi_fifo_push writes the len byte + payload then shm->midi_write = wr at 783 with no preceding fence. The code DOES use __sync_synchronize() for pending_params (798/1026), pending_single (1069/2095) and preset_req/done_gen (1182/2141/2683) — so the author knows the idiom — but the two hot ring buffers are unfenced. On Move's ARM Cortex-A72 (weak memory model, producer and consumer always on different cores/processes), the sample/payload stores can become visible AFTER the index store, letting the consumer read stale/torn ring slots: audible audio glitches and occasional corrupted MIDI bytes (e.g. a mangled note or program-change) under store-buffer pressure. Reachable but intermittent and hardware-timing-dependent, hence MEDIUM. Fix: __sync_synchronize() (or a release store) before publishing ring_write/midi_write in the child, and an acquire (__sync_synchronize after reading the index) in the parent before touching the data.

_Probe:_ Add a release fence before the ring_write store (child emu loop + prefill) and before the midi_write store in midi_fifo_push; add an acquire fence in shm_ring_available/v2_render_block and child_process_midi_fifo after loading the published index. Confirm the prefill path (1580) and reset (1763) are consistent.

### G3. Multi-producer MIDI FIFO without a lock (LOW/conditional, verify host threading)

midi_fifo_push (773-784) is a non-atomic single-producer primitive (read midi_write -> write bytes -> store midi_write) but has TWO distinct callers on potentially different host threads: the real-time MIDI path v2_on_midi (1920, 1950) and the control path v2_set_param (preset/bank switches push CC32+PC at 2143-2219, all_notes_off at 2266). If those two host callbacks can run concurrently, both producers read the same midi_write, write overlapping slots, and one store clobbers the other -> lost/corrupted MIDI (e.g. a preset switch racing an incoming note could drop the Program Change, stranding the old patch, or inject a malformed status byte). The schwung docs indicate MIDI filtering and audio mixing both run in the SPI ioctl callback (ARCHITECTURE.md:82) and that param/load work also reaches the DSP from the SPI thread (REALTIME_SAFETY.md:112-118), which would serialize all of these on one thread and make the race impossible. But this is NOT guaranteed by the plugin API and not asserted anywhere in the module. Severity is LOW *if* serialized, MEDIUM if remote-UI set_param can land on a separate thread from on_midi. Worth confirming the host's delivery-thread guarantee for on_midi vs set_param (especially manager/shadow_set_param-originated calls).

_Probe:_ Confirm in the host (schwung_shim / shadow_ui delivery) whether v2_on_midi and v2_set_param for a given slot are always dispatched on the same (SPI) thread. If not, guard midi_fifo_push with a tiny spinlock or move set_param-originated MIDI through a separate single-producer channel.
